## Story

**Como o** sistema meDIZ,
**eu preciso** de um mecanismo que detecte, sem depender de um "fim de conversa" que não existe no produto, quando uma fatia nova de conversa está pronta para ser processada,
**para que** o ID Vida (e futuramente a atualização contínua do ID Pessoa) tenham um outbox confiável de onde consumir trechos de conversa, sem cron de inatividade e sem reprocessar o que já foi extraído.

## Acceptance Criteria

1. Novo model `ConversationEvent` em `prisma/schema.prisma` — outbox genérico e compartilhado (não exclusivo do ID Vida; desenhado para também alimentar, numa story futura, a atualização contínua do ID Pessoa — `core`/`dynamics`/`predictive`), copiando a estrutura de `DiscoveryEvent`: `id`, `userId`, `sessionId`, `eventType` (default `'conversation.checkpoint.v1'`), `schemaVersion`, `payload` (Json), `status` (default `'pending'`), `attempts`, `lastError`, `availableAt`, `processedAt`, `createdAt`, `updatedAt` — mais um campo novo, `trigger` (`String`, CHECK constraint `'message_count' | 'new_session'`). Migration aditiva manual (sem shadow database — banco compartilhado com `indevmediz`, checar `prisma migrate status` antes de aplicar, mesmo procedimento da Story 2.1/2.2).
2. `ChatSession` ganha dois campos novos: `messageCount Int @default(0)` e `lastCheckpointMessageCount Int @default(0)`, na mesma migration.
3. `saveChatMessage` (`src/lib/chatMessages.ts`) passa a rodar dentro de uma `prisma.$transaction` que: (a) cria a `ChatMessage` como hoje; (b) incrementa `ChatSession.messageCount`; (c) se `messageCount - lastCheckpointMessageCount >= CONVERSATION_CHECKPOINT_MESSAGE_THRESHOLD` (env var nova, default `14`), insere um `ConversationEvent` (`trigger: 'message_count'`) e atualiza `lastCheckpointMessageCount = messageCount` no mesmo `update` — para não dobrar o disparo na mesma janela.
4. `createChatSessionWithThread` (`src/lib/chatService.ts`) passa a, antes de criar a sessão nova, buscar a sessão mais recente do usuário com `messageCount > lastCheckpointMessageCount` (ou seja, com conteúdo não extraído) e, se existir, enfileirar um `ConversationEvent` (`trigger: 'new_session'`) para ela.
5. `POST /api/conversation-events/claim` e `POST /api/conversation-events/result`, copiando o padrão de `src/app/api/discovery/events/{claim,result}/route.ts` (SQL raw com `FOR UPDATE SKIP LOCKED`, até 5 eventos por chamada, eventos presos em `processing` por mais de 10 min voltam para `failed`, retry com backoff exponencial até 60 min). Autenticados via `validateWebhookBearer(request, 'CONVERSATION_WEBHOOK_TOKEN')` — env var própria e dedicada, reusando a função já generalizada em `src/lib/webhookAuth.ts` (não criar um 3º padrão de bearer auth).
6. `payload` do `ConversationEvent` carrega só a referência à fatia nova da conversa desde o último checkpoint (ex.: `{ threadId, fromMessageCount, toMessageCount }`), não o conteúdo das mensagens em si — quem processar o evento (Story 4.3, fora deste repo) busca o texto via API própria quando precisar. Mantém o outbox leve e evita duplicar dado sensível em duas tabelas.
7. Enquanto `saveChatMessage` está sendo alterado por esta story, remover as duas chamadas `fetch()` de debug hoje hardcoded na função (`http://127.0.0.1:7242/ingest/...`, com `contentPreview` do conteúdo da mensagem) — instrumentação de depuração esquecida em commit anterior, sem relação com este código, que hoje roda em produção a cada mensagem de chat.
8. **Fora de escopo desta story**: qualquer extração de conteúdo (fatos do ID Vida ou perfil do ID Pessoa) — só os dois gatilhos de disparo e o enfileiramento. Também fora de escopo: cron de inatividade para sessões curtas abandonadas (ver limitação aceita no plano — vira Story 4.5 só se os dados mostrarem que é necessário).

- [ ] Envolver `saveChatMessage` numa transação que cria a mensagem, incrementa `messageCount`, e condicionalmente enfileira o checkpoint por contagem (AC: 3).

## Dev Notes

- **Fonte deste plano**: `docs/plano-id-vida-implementacao.md`, seção "Story 4.1 — Gatilhos de checkpoint (outbox compartilhado)" e seção "Gap crítico descoberto" (justifica por que o gatilho é checkpoint incremental, não "fim de conversa").
- **Padrão a copiar literalmente**: `src/app/api/discovery/events/claim/route.ts` e `.../result/route.ts` (query SQL raw com `FOR UPDATE SKIP LOCKED`, contagem de tentativas, backoff) — mesmo problema resolvido de novo, não inventar uma solução nova. Confirmado presente em `origin/develop` no momento desta story.
- **`validateWebhookBearer`** (`src/lib/webhookAuth.ts`) já aceita nome de env var customizado como segundo parâmetro (generalizada na Story 2.1) — usar `CONVERSATION_WEBHOOK_TOKEN` como a env var dedicada deste outbox, sem tocar na função.
- **Threshold `X` (revisão @architect)**: comece com `14` (meio da faixa 12–15 sugerida no plano original), mas trate como parâmetro tunável por dado real — não há baseline de tamanho médio de conversa neste app ainda. Deixar configurável por env var (`CONVERSATION_CHECKPOINT_MESSAGE_THRESHOLD`) em vez de hardcoded, exatamente para permitir recalibrar sem deploy de código depois de medir em produção.
- **Contrato do `payload`/endpoint `result` (decisão pendente resolvida na revisão @architect)**: como o outbox é compartilhado entre o braço ID Vida (Story 4.3) e uma futura story de atualização contínua do ID Pessoa, o endpoint `result` desta story deve aceitar um payload de resultado que **não assuma um único consumidor** — ou seja, não travar o schema do `result` em campos exclusivos de `UserFact`/`lifeCompact`. Nesta story (4.1), o endpoint `result` só precisa fazer o básico do outbox (marcar `processed`/`failed`, gravar `lastError`, agendar retry); a escrita específica de dados de perfil/fatos fica para as stories que de fato processam o conteúdo (4.3 e a futura story do ID Pessoa). Documentar essa divisão explicitamente no código do endpoint (comentário curto basta) para quem for abrir a Story 4.3 não presumir um contrato mais amplo do que este.
- **Achado de arquitetura não relacionado, mas tratado nesta story por tocar o mesmo arquivo**: `saveChatMessage` em `src/lib/chatMessages.ts` (em `develop`) tem hoje dois `fetch()` incondicionais para `http://127.0.0.1:7242/ingest/87541063-b58b-4851-84d0-115904928ef7`, enviando `contentPreview` (100 primeiros caracteres da mensagem) a cada mensagem salva — parece instrumentação de debug de um agente de IA (`hypothesisId`, `runId`, `sessionId: 'debug-session'`) esquecida num commit anterior. Remover como parte do AC 7, já que a função inteira está sendo reescrita para virar uma transação.
- **Assunção de ordenação**: `fromMessageCount`/`toMessageCount` no payload assume que `ChatMessage` nunca é deletada fora de ordem dentro de uma sessão (não há `delete` de `ChatMessage` encontrado no código hoje) — se isso mudar no futuro, o cálculo da fatia por contagem para de ser confiável e precisa virar referência por `id`/`createdAt`.
- **Banco compartilhado**: como descoberto na Story 2.1, o Postgres do meDIZ é compartilhado com o projeto legado `indevmediz`. Checar `prisma migrate status` antes de qualquer `migrate dev`/`deploy`.
- **Branch**: esta story deve ser implementada a partir de `feat/id-vida-checkpoint-outbox` (criada de `origin/develop` em 2026-07-18) — `main` não tem nenhuma das peças do ID Pessoa/Discovery das quais esta story depende (`UserProfile`, `DiscoveryEvent`, `validateWebhookBearer` generalizada). Confirmar que a branch de trabalho não é baseada em `main` antes de começar.
