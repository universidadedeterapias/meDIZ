## Story

**Como uma** pessoa que acabou de confirmar o WhatsApp no cadastro,
**eu quero** ser levada automaticamente para a descoberta antes do chat,
**e como um** admin,
**eu quero** ver o perfil extraido de cada usuario,
**para que** a descoberta vire parte natural do onboarding e a equipe tenha visibilidade do que foi captado.

## Dependencies

- Depende da Story 2.1 (endpoints e feature flag).
- Depende da Story 2.3 (a tela `/descoberta` precisa existir para o redirecionamento fazer sentido).
- Depende da Story 2.2 para o admin mostrar dados reais (sem ela, a secao aparece vazia/pendente — nao e bloqueante, so incompleta).

## Acceptance Criteria

1. Ao carregar `/chat` autenticado (mesmo ponto onde hoje existe a checagem `checkingProfile`), se `shouldRunDiscovery` indicar que o usuario e elegivel e ainda nao completou (`discoveryCompleted=false`), redireciona para `/descoberta` antes de mostrar qualquer mensagem/tela de chat.
2. O gate roda **depois** da verificacao de WhatsApp existente (fluxo de cadastro atual nao muda) — descoberta e a primeira coisa que acontece na primeira visita autenticada ao chat, nao antes disso.
3. Usuarios com `discoveryCompleted=true` (incluindo todos os backfillados na Story 2.1) nunca veem o redirecionamento.
4. A tela de detalhe de usuario em `/admin/users` ganha uma secao nova mostrando o perfil de descoberta quando existir: `usageContext`, `preferredStyle`, `compactProfile`, e os tracos `core`/`dynamics`/`predictive` (formato legivel, nao JSON cru). Se a extracao (Story 2.2) ainda nao rodou, mostra um estado "processamento pendente" em vez de erro ou secao vazia confusa.
5. Nenhuma quebra para usuarios sem `UserProfile` (usuarios pre-existentes podem nao ter registro algum, dado que a descoberta nunca rodou para eles) — a secao admin trata a ausencia graciosamente.
6. Feature flag `DISCOVERY_ENABLED` continua controlando se o gate roda de fato — por padrao permanece desligada ate validacao manual em produção, mesma cautela do repo de origem.
